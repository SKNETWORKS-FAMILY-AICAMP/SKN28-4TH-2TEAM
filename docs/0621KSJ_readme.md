## 1. 인수인계 5개 의심 버그 — 검증 결과

| # | 주장 | 판정 | 근거 |
|---|---|---|---|
| P1-a | 질문이 2회 분석됨 | **무해(격하)** | `analyze()`는 순수함수라 2번 호출해도 결과 동일. 성능 비용도 무시 가능 |
| P1-b | 근거검사(`_has_direct_evidence`)가 일반/학과소개를 우회 | **무해 → drop** | 우회를 제거해도 no-op(검색기가 이미 학과 필터). 실제 소개 답변도 양호 |
| P2 | 분류기가 너무 많은 책임을 가짐 | 사실(리팩터링 가치 O) | 미착수(유지보수용, 선택) |
| P3 | 선착순 키워드 분류로 오분류 | **사실 → 버그 A·B** | 단, 원인은 "키워드 *위치*"가 아니라 "룰 *순서* + 단일 의도 한계"였음 |
| P4 | PDF 청킹이 글자수 기준 | 후순위(데이터 과제) | AX엔 서술(text) 문서가 0개 등 타이핑 불일치 확인. 보류 |

> **중요한 발견**: P1-a, P1-b는 "중간 신호"(분석 호출 횟수, 의도 라벨, content_type 라벨)로 보면 버그처럼 보이지만, **실제 생성 답변을 보면 멀쩡**했다. 반대로 진짜 버그(A·B)는 실제 답변을 생성해봐야 드러났다.

### 의심버그 최종 상태 (해결 / 애매 / 미해결)

**✅ 명확히 해결**
- **P3 (선착순 키워드 분류)** — 실제 사용자 결함 2개로 구체화되어 둘 다 수정 + 답변레벨 검증:

  - **버그 B** 전체 학과 목록 불완전(4개 중 2개·사실오류) → scope 4학과 커버리지로 해결.
  - 재검토에서 발견한 과잉트리거("입학 일정"류)도 접속어 게이트로 추가 해결.

**🟡 애매 — "버그처럼 보였으나 무해"로 판명(미수정)**
- **P1-a (질문 2회 분석)** — `analyze()`가 순수함수라 결과 동일. 고칠 필요 없음으로 판명(미수정).
- **P1-b (근거검사 우회)** — 우회를 제거해도 no-op(검색기가 이미 학과 필터)이고 실제 소개 답변도 양호 → drop.
- ※ 둘 다 "고친 것"이 아니라 "고칠 필요가 없음을 입증한 것"이다.

**❌ 미해결 / 보류**

## 4. 의사결정 기록 (왜 이렇게 했는가)

> 이 작업은 갈림길이 많았다. "쉬운 길"이 아니라 "데이터/실측이 지지하는 길"을 고르려고 했고, 그 과정에서 몇 번 방향을 바꿨다. 팀이 나중에 "왜 이렇게 했지?"를 이해할 수 있도록 기록한다.

### D1. 작업 방식: 추측 수정 vs 검증 후 수정 → **검증 후**
인수인계 코멘트가 전부 "~인 것 같다" 톤이라, 각 주장을 코드/실측으로 참·거짓 판정한 뒤에만 손댔다. 그 결과 의심 버그 5개 중 3개가 무해로 판명되어 **불필요한(심지어 유해한) 수정을 피했다.**

### D2. P1-a(분석 2회): 긴급 버그 vs 무해 청소 → **무해로 격하**
파이프라인과 VectorRetriever가 `analyze()`를 각 1회씩 부르는 건 사실. 그러나 `analyze()`는 (질문, 직전학과)의 순수함수라 두 결과가 항상 동일 → 정합성 버그 아님. "성능 버그"라는 인수인계 프레이밍은 과장이었다. 우선순위 최하로 내림.

### D3. S3 근거게이트: 핵심 의사결정 (여러 번 방향 전환) → **결국 DROP**
- 처음엔 "일반/학과소개도 근거검사를 거치게 하자"는 인수인계 지적을 받아 **등급제 게이트**를 설계하려 했다.
- 사용자가 "쉬운 길 말고 올바른 길"을 요구 → 데이터를 직접 확인.
- **발견 1**: "학과 소개" 전용 content_type이 데이터에 없음(소개/overview 문자열 포함 문서 778개 중 2개) → 콘텐츠타입 기준(B안)은 **구현 불가**(과차단).
- **발견 2**: 검색기가 특정 학과 질문이면 이미 그 학과 문서만 통과시킴 → 학과-스코프 게이트(A안)는 **no-op**.
- **발견 3**: 실제로 학과 소개 답변을 생성해보니, content_type이 admission/course로 라벨링됐어도 내용은 **PDF 서술 기반으로 양호**했다. 즉 "소개가 조각으로 만들어져 나쁘다"는 전제 자체가 틀림.
- **결론**: 근거게이트 수정은 no-op이거나 거짓거부를 유발 → **S3와 그 대안(검색가점 S3′) 모두 폐기.** (P1-a처럼 "버그처럼 보였지만 무해")

### D4. S5 분류 엔진: 순수규칙 점수제 vs LLM vs 하이브리드 → **순수규칙(결정적), 격상은 측정 조건부**
- LLM 분류기는 의미 이해엔 강하나 비결정적이라 **오프라인 검증 하네스를 깨고** 매 질문 지연·비용·인젝션 표면을 늘림.
- 측정된 오분류는 전부 결정적 규칙으로 해결 가능했고, LLM이 *반드시* 필요한 의미 실패는 측정되지 않음.
- → 1차는 결정적 규칙. 임베딩/LLM 격상은 "골든이 잔여 의미오류를 숫자로 보여줄 때"만. (검증 안 된 단계에서 LLM 박는 것도 게으름의 반대 버전)

### D5. 다중 의도 범위: (a) 최소 2-의도 vs (b) N-일반 → **모델·배관은 N-일반 한 번에, 분석기 채우기는 보수적**
- (a)를 2-의도 특수 케이스로 하드코딩하면 나중에 (b)에서 통째로 들어내야 함(재작업 2번).
- 변경 비용의 대부분은 "스칼라→집합" 배관 공사인데 그건 2개든 N개든 한 번 들어감.
- → **데이터모델·배관은 N-일반으로 한 번에**, 분석기는 명확한 공동 의도만 보수적으로 채움. 나중에 일반화(b)는 **분석기 한 곳만** 손보면 됨(재작업 0).

### D6. 골든 큐레이션: 코드에 맞춰 통과시키기 vs 근거별 판정 → **근거별 판정**

### 8.0 발단: 노트의 "46명 표"가 거짓이었다
- 재베이스라인: `validate_rag_quality.py` **40/40**, `sql_multi_intent_smoke.py` **ALL PASS**, 버그A(`@` 포함)·버그B(4학과) 답변레벨 재확인 — 여기까진 노트와 일치.

### 8.1 원인 추적 — 가설을 측정이 두 번 뒤집음
1. **단계별 행수 계측**: `people.csv 46` = `SQL raw 46` = **`built_context 30`** = `answer 30`. → 절단 지점은 LLM이 아니라 **context_builder**(LLM은 받은 30을 충실히 렌더, `answer ⊆ context` 성립).

## 8.6 (세션3, 2026-06-21) 거짓 총계 · 예산 불공정 해결 — "S7"

> 8.4에서 "멈추고 한 번 더"로 도출했던 **미해결 ①②**를 코드로 닫은 기록. 콜드스타트로 8.4 착수점에서 이어감. 두 수정 모두 [[feedback_verify_at_output_level]]대로 **실제 컨텍스트 고지 문자열·섹션 길이**로 판정(MySQL·CSV 양쪽), 결정적 스모크(#43/#44)로 고정.

### ① 거짓 총계 → 진짜 COUNT (해결)
- **원인 재확인**: 절단 고지 `총 N개`의 `N = len(sql_result.rows)`인데 rows는 `LIMIT 100` **이후** 값. AX person 실제 **147명**인데 고지가 "총 100개"로 거짓 단정.
- **수정**: `SqlQueryResult`에 `total_available`(LIMIT 전 진짜 매칭 수, None=미상→len(rows) 후퇴) 추가. **MySQL**은 `COUNT(*) OVER() AS _total_count` 윈도우 함수로 단일 쿼리에서 캡 전 총계 획득 후 `_split_total_count`로 행에서 분리(표/컬럼 누출 0). **CSV**는 `.head(limit)` 전 `len()`. generic/person·course·asset 경로 + 양쪽 일치. context_builder는 고지 `total = max(total_available, len(rows))`.
- **보너스(예고대로 동시 해결)**: "SQL캡 침묵". course(CSV 145)는 이전엔 displayed=100=len(rows)라 `displayed<total`이 false → **고지 자체가 없었음**. 이제 "총 145개 중 100개만 표시"로 캡 사실이 드러남.
- **before→after**(AX person 147): 고지 `총 100개`(거짓, MySQL·CSV 공통) → `총 147개`(정직). `_total_count` 컬럼 비누출 확인.

### ② 예산 불공정 → 공정 분할 (해결)
- **결정(D17)**: "우선순위 정렬" 대신 **예산 균등 분할** 채택. ①로 고지가 정직해진 지금 배분의 임무는 "어느 표가 중요한지 맞히기"가 아니라 "요청한 각 항목을 공정 미리보기"다. 우선순위 정렬은 둘 다 거대하면(person147·course145) 굶주림을 **뒤집을 뿐 해결 못 함** + 의도파싱이라는 깨지기 쉬운 로직 추가. 균등 분할은 구조적으로 굶주림 차단·순서 무관.
- **수정**: `_build_sql_contexts_fair` — **작은 표(행수)부터** `남은예산 // 남은표수`로 할당(작은 표가 남긴 몫이 큰 표로 흐름), **표시는 원래 순서 유지**(배분 공정성 ↔ 표시순서 분리). 빈 결과는 먼저 렌더해 공정 몫에서 제외.
