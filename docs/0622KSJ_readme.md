## 3. 세션별 진행 (S5~C)
- **S5 다중의도 정교화**: 모호 키워드 가드(홈페이지→person, 입학"일정"→event 오귀속 차단). 그 과정에서 **내가 만든 출력단계 회귀(면접 일정 행 누락)** 를 발견·수정, CSV/MySQL `suppress` 미적용 선재 버그도 교정. → 분류 40/40, SQL 스모크 추가.
- **S6 교수목록 완전성**: 노트의 "교수 이메일 46명 표"가 **거짓**(실제 렌더 30행)임을 답변 전문으로 확인. 원인은 행 캡이 아니라 **컬럼 부풀림**(`SELECT *` 18컬럼이 12000자 문자캡 초과). → 컬럼 프루닝·행캡 정합·행경계 절단+고지·고지 답변반영·단일의도 person 0행 수정. 컨텍스트 46/46 복원.
- **S7 거짓 총계 · 예산 불공정**(S6 재검토에서 도출한 미해결 ①②):
  - **① 거짓 총계** → `total_available`(LIMIT 전 진짜 COUNT) 도입. MySQL은 `COUNT(*) OVER()`, CSV는 `.head()` 전 `len()`. "총 100개"(거짓) → "총 147개"(정직). "SQL캡 침묵"도 동시 해결.
  - **② 예산 불공정** → **균등 분할**(`_build_sql_contexts_fair`, 작은 표부터 할당, 표시순서 유지). person/course 섹션 문자 길이 min/max=0.99.
  - 검증: 분류 40/40, 스모크 #37~#44 ALL PASS, **최종 LLM 답변까지** "총 147명" 확인.
- **C 회귀가드 강화 + 형제 버그 감사**:
  - **C-1**: `total_available` 가드를 course·asset 경로로 확장(#45/#46). 오라클 = SQL 캡을 10만으로 올린 uncapped 행수와 대조(데이터값 하드코딩 없음). #37~#46 ALL PASS.
  - **C-2**: "post-LIMIT 행수를 총계처럼 사용"하는 형제 버그를 소비처까지 grep 추적 → **CLEAN(미발견)**. `row_count`는 생성만 되고 Streamlit이 미소비.

## 7. 작업 방식 교훈
1. 진단은 **증상과 원인을 분리 검증** (P3 "위치설" → 실제는 "룰 순서 + 단일 의도 한계").
2. **자명한 한 줄 수정이 no-op일 수 있다** (P1-b).
3. **버그 판정은 최종 답변으로** — 추론으로 내린 결론이 실측에서 반복 반전됐다.
4. **검증 통과 ≠ 올바름** — "멈추고 한 번 더"가 S7의 미해결 ①②를 잡았다.
5. **저위험은 착수 신호가 아니다** — 가치 검증 후의 부차 조건일 뿐 (이미 충실한 README를 덮어쓰지 않음).
