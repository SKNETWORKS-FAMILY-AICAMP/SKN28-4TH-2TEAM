## 1. 테스트 목적

본 테스트는 KAIST AI 학과 안내 RAG 챗봇 웹 서비스가 다음 품질 기준을 만족하는지 구간별로 확인하기 위해 수행했다.

- Django 웹 애플리케이션이 정상적으로 로드되고 주요 API가 동작하는지 확인한다.
- 회원가입, 로그인 상태, 게시판, 문의, 채팅 세션 등 웹 서비스 핵심 흐름을 검증한다.
- **질문 입력 → API 요청 → 라우팅/분류 → 답변 응답 출력**으로 이어지는 핵심 기능 흐름을 검증한다.
- RAG 질문 분석이 골든셋 기준 route, intent, department, content type을 정확히 분류하는지 확인한다.
- SQL/Context 회귀 테스트로 다중 의도 질문 처리와 context 구성 안정성을 확인한다.
- **빈 입력, 길이 초과, 잘못된 요청, 응답 실패, 답변 불가** 등 예외 상황에서 서버가 무중단으로 안전하게 응답하는지 확인한다.
- 데이터 파일, VectorStore, 기존 RAG 평가 산출물의 준비 상태를 확인한다.
- 외부 LLM/API 의존 구간과 수동 UI 검증 구간은 별도 리스크로 분리한다.

주요 확인 사항:

- 직접 retrieval 평가는 query embedding 생성 단계에서 `APIConnectionError: Connection error`가 발생하여 hit@5 지표를 산출하지 못했다.
- 파이프라인 평가는 30문항 모두 실행되었으나, 대부분 SQL 조회 결과 부족 또는 vector fallback 실패로 인해 "제공된 자료에서 질문에 대한 충분한 근거를 찾을 수 없습니다" 유형의 응답을 반환했다.
- 자연과학대학 1문항은 SQL route로 진입했으나 답변 생성 단계에서 connection error가 발생하여 생성 오류로 분류했다.

- `event` 데이터에서 일부 학과 조건 조회 결과 없음

결론:

신규 데이터는 VectorStore에 저장되어 있으나, 현재 실행 환경에서는 OpenAI embedding 및 답변 생성 API 연결 오류로 인해 RAG 검색 및 생성 품질을 정상적으로 검증하지 못했다. 이번 결과는 신규 데이터 미적재 문제가 아니라, RAG 실행 의존성(API 연결)과 SQL fallback coverage 부족이 결합된 실패 결과로 판단한다. API 연결을 정상화한 뒤 동일 평가를 재실행해야 신규 데이터 기준 retrieval hit@5, keyword hit@5, 정상 답변률을 다시 산출할 수 있다.

## 7. 핵심 기능 End-to-End 검증 (질문 입력 → API 요청 → LLM 응답 출력)

평가 기준 2번("질문 입력, API 요청, LLM 응답 출력 등 핵심 기능이 실제로 검증되었는가")에 대응하여, 채팅 핵심 기능을 흐름 단계별로 검증한 결과를 정리한다. 코드 경로는 [chat/views.py](../src/kaist_rag/apps/chat/views.py) → [chat/services.py](../src/kaist_rag/apps/chat/services.py) → [rag/rag_pipeline.py](../src/kaist_rag/rag/rag_pipeline.py)이다.

### 7.1 흐름 단계

```
사용자 질문 입력
   │  (UI: chat.js, Enter 전송)
   ▼
POST /api/chat/  ──▶ _clean_question() 정규화 ──▶ _validate_question() 입력 검증
   │                                                  └─ 빈 입력/1000자 초과 시 400
   ▼
answer_question(question, previous_department_code)
   │  ├─ get_pipeline()  (RAG_ENABLE_ENGINE=false면 RuntimeError → fallback)
   │  └─ pipeline.run()  ─▶ QueryAnalyzer(route/intent/department/content_type 분류)
   │                       ─▶ SQL Tool / Vector Retriever (하이브리드 검색)
   │                       ─▶ AnswerGenerator (LLM 답변 생성)
   ▼
ChatAnswer(answer, sources, warning, route) ──▶ DB 저장(ChatMessage) ──▶ JSON 응답 출력
   │
   └─ 예외 발생 시: FALLBACK_ANSWER + warning="RAG_UNAVAILABLE" + route="blocked" (서버 무중단)
```

### 7.2 단계별 검증 결과

| TC ID | 단계 | 검증 방법 | 기대 결과 | 실제 결과 | 판정 | 비고 |
|---|---|---|---|---|---|---|
| E2E-01 | 질문 입력 정규화 | `_clean_question`이 앞뒤 공백/중복 공백 정리 | 정규화된 질문 문자열 | 코드 경로 확인 (`" ".join(...split())`) | Pass | [views.py:28](../src/kaist_rag/apps/chat/views.py#L28) |
| E2E-02 | 입력 검증 | 빈 질문/길이 초과 시 400 반환 | 400 + 안내 메시지 | F-16(빈 입력) Pass, 길이 초과는 8장 EX-02 | Pass | [views.py:32](../src/kaist_rag/apps/chat/views.py#L32) |
| E2E-03 | API 요청 수신 | `POST /api/chat/` (`require_http_methods(["POST"])`) | 정상 라우팅, 비POST 차단 | Django test client로 POST 검증 | Pass | F-17 |
| E2E-04 | 질문 분석/분류 | `QueryAnalyzer`가 route/intent/department/content_type 분류 | 골든셋 기대값 일치 | D-01: 40/40 통과 | Pass | LLM 비의존 |
| E2E-05 | 하이브리드 검색 | SQL Tool + Vector Retriever 조합으로 근거 행/문서 구성 | 의도별 SQL 행·context 정직 구성 | E: 13개 중 10 Pass (3건은 테스트 전제) | ⚠️ Partial | E 구간 참고 |
| E2E-06 | LLM 답변 생성 출력 | 실제 OpenAI 기반 답변 생성 후 JSON 출력 | 출처 포함 정상 답변 | H: 30문항 정상 답변 0건 (API 연결 오류) | Fail | RISK-001 |
| E2E-07 | 응답 페이로드 구조 | 응답에 `answer/sources/warning/route/session_id` 포함, DB 저장 | 필드 정상 반환 및 세션·메시지 영속화 | F-17~F-19 Pass (fallback 경로) | Pass | 세션 1개, 메시지 2개 |

> 정리: 핵심 기능 흐름 중 **질문 입력 → 입력 검증 → API 요청 → 분류/라우팅 → 응답 페이로드/세션 영속화**까지는 검증을 완료(Pass)했다. 다만 **실제 LLM 답변 생성 출력(E2E-06)** 은 OpenAI API 연결 오류로 정상 답변을 확인하지 못했으며, 이는 11장 RISK-001로 추적한다. RAG 비활성 상태에서도 서버가 fallback 응답으로 무중단 동작함은 F-17로 확인했다.

| 개선 항목 | 이유 | 우선순위 |
|---|---|---|
| 관리자 화면 권한 보호 | 운영 통계는 관리자 기능이므로 직접 URL 접근 차단 필요 | High |
| OpenAI API 연결 설정 점검 | Embedding/LLM 연결 오류가 발생하면 VectorStore가 있어도 RAG 답변 품질을 검증할 수 없음 | High |
| SQL 회귀 테스트 전제 고정 | 설정값 변경으로 회귀 테스트가 깨지지 않도록 테스트 fixture/환경값 분리 필요 | High |
| SQL fallback 학과별 coverage 보완 | 일부 학과의 person/course/admission/event 조회 결과가 없어 vector fallback 실패 시 근거 부족 응답으로 이어짐 | High |
| 예외 케이스 자동 테스트 추가 | EX-02/03/04/07를 자동 스모크에 포함해 회귀 방지 | Medium |
| RAG 품질 평가 자동 보고 | `rag_eval_results_*.json`을 보고서 표로 자동 변환하면 품질 추적이 쉬움 | Medium |
| 웹 E2E 자동화 | 로그인→채팅→출처→게시판→문의 흐름을 반복 검증 가능 | Medium |
| UI 반응형 캡처 | 발표 전 화면 깨짐/겹침 리스크 감소 | Low |
