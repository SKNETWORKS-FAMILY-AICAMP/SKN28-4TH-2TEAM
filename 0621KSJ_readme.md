## 1. 인수인계 5개 의심 버그 — 검증 결과

| # | 주장 | 판정 | 근거 |
|---|---|---|---|
| P1-a | 질문이 2회 분석됨 | **무해(격하)** | `analyze()`는 순수함수라 2번 호출해도 결과 동일. 성능 비용도 무시 가능 |

> **중요한 발견**: P1-a, P1-b는 "중간 신호"(분석 호출 횟수, 의도 라벨, content_type 라벨)로 보면 버그처럼 보이지만, **실제 생성 답변을 보면 멀쩡**했다. 반대로 진짜 버그(A·B)는 실제 답변을 생성해봐야 드러났다.

  - 재검토에서 발견한 과잉트리거("입학 일정"류)도 접속어 게이트로 추가 해결.

**🟡 애매 — "버그처럼 보였으나 무해"로 판명(미수정)**
- **P1-a (질문 2회 분석)** — `analyze()`가 순수함수라 결과 동일. 고칠 필요 없음으로 판명(미수정).
- **P1-b (근거검사 우회)** — 우회를 제거해도 no-op(검색기가 이미 학과 필터)이고 실제 소개 답변도 양호 → drop.
- ※ 둘 다 "고친 것"이 아니라 "고칠 필요가 없음을 입증한 것"이다.

**❌ 미해결 / 보류**
- **P2 (분류기 비대)** — 사실. 다중속성 도입으로 일부 정돈됐으나 **책임분리 리팩터링 자체는 미착수**(사용자 체감 0·회귀 위험 → 보류).
- **P4 (PDF/청킹)** — 미착수(데이터·청킹 과제, 후순위).
- **버그 B의 '내용 풍부도'** — 완전성(4학과 표기)은 고쳤으나 학과별 소개 *내용*이 빈약(예: AX는 course 위주)한 건 데이터/P4 과제로 미해결.

## 4. 의사결정 기록 (왜 이렇게 했는가)

> 이 작업은 갈림길이 많았다. "쉬운 길"이 아니라 "데이터/실측이 지지하는 길"을 고르려고 했고, 그 과정에서 몇 번 방향을 바꿨다. 팀이 나중에 "왜 이렇게 했지?"를 이해할 수 있도록 기록한다.

### D1. 작업 방식: 추측 수정 vs 검증 후 수정 → **검증 후**
인수인계 코멘트가 전부 "~인 것 같다" 톤이라, 각 주장을 코드/실측으로 참·거짓 판정한 뒤에만 손댔다. 그 결과 의심 버그 5개 중 3개가 무해로 판명되어 **불필요한(심지어 유해한) 수정을 피했다.**

### D2. P1-a(분석 2회): 긴급 버그 vs 무해 청소 → **무해로 격하**
파이프라인과 VectorRetriever가 `analyze()`를 각 1회씩 부르는 건 사실. 그러나 `analyze()`는 (질문, 직전학과)의 순수함수라 두 결과가 항상 동일 → 정합성 버그 아님. "성능 버그"라는 인수인계 프레이밍은 과장이었다. 우선순위 최하로 내림.

### D3. S3 근거게이트: 핵심 의사결정 (여러 번 방향 전환) → **결국 DROP**
- 처음엔 "일반/학과소개도 근거검사를 거치게 하자"는 인수인계 지적을 받아 **등급제 게이트**를 설계하려 했다.
- 사용자가 "쉬운 길 말고 올바른 길"을 요구 → 데이터를 직접 확인.
- **발견 1**: "학과 소개" 전용 content_type이 데이터에 없음(소개/overview 문자열 포함 문서 778개 중 2개) → 콘텐츠타입 기준(B안)은 **구현 불가**(과차단).
- **발견 2**: 검색기가 특정 학과 질문이면 이미 그 학과 문서만 통과시킴 → 학과-스코프 게이트(A안)는 **no-op**.
- **발견 3**: 실제로 학과 소개 답변을 생성해보니, content_type이 admission/course로 라벨링됐어도 내용은 **PDF 서술 기반으로 양호**했다. 즉 "소개가 조각으로 만들어져 나쁘다"는 전제 자체가 틀림.
- **결론**: 근거게이트 수정은 no-op이거나 거짓거부를 유발 → **S3와 그 대안(검색가점 S3′) 모두 폐기.** (P1-a처럼 "버그처럼 보였지만 무해")

### D4. S5 분류 엔진: 순수규칙 점수제 vs LLM vs 하이브리드 → **순수규칙(결정적), 격상은 측정 조건부**
- LLM 분류기는 의미 이해엔 강하나 비결정적이라 **오프라인 검증 하네스를 깨고** 매 질문 지연·비용·인젝션 표면을 늘림.
- 측정된 오분류는 전부 결정적 규칙으로 해결 가능했고, LLM이 *반드시* 필요한 의미 실패는 측정되지 않음.
- → 1차는 결정적 규칙. 임베딩/LLM 격상은 "골든이 잔여 의미오류를 숫자로 보여줄 때"만. (검증 안 된 단계에서 LLM 박는 것도 게으름의 반대 버전)

### D5. 다중 의도 범위: (a) 최소 2-의도 vs (b) N-일반 → **모델·배관은 N-일반 한 번에, 분석기 채우기는 보수적**
- (a)를 2-의도 특수 케이스로 하드코딩하면 나중에 (b)에서 통째로 들어내야 함(재작업 2번).
- 변경 비용의 대부분은 "스칼라→집합" 배관 공사인데 그건 2개든 N개든 한 번 들어감.
- → **데이터모델·배관은 N-일반으로 한 번에**, 분석기는 명확한 공동 의도만 보수적으로 채움. 나중에 일반화(b)는 **분석기 한 곳만** 손보면 됨(재작업 0).

### D6. 골든 큐레이션: 코드에 맞춰 통과시키기 vs 근거별 판정 → **근거별 판정**
"코드 출력대로 골든을 바꿔 통과시키기"(teaching to test)는 거부. 11개 실패를 (가)낡음 / (나)실버그 / (제품결정)으로 분류해, 낡은 건 근거 있게 정정하고, 진짜 결함은 그대로 두고, 입학 라우팅(sql-first vs vector)은 답변 품질 측정 전까지 **OPEN**으로 보류.

### D7. 버그 B 원인: 라우팅 vs 검색커버리지 vs 데이터 → **검색커버리지로 수정, 내용은 P4 보류**
- 처음엔 #5 라우팅(sql vs hybrid) 문제로 의심.
- 추적: SQL `department` 테이블이 비어 둘 다 vector로 답함 → 라우팅 무관.
- 진짜 원인: scope 질문의 vector 커버리지(전역 검색이 AX를 놓침).
- 학과별 검색으로 **완전성(4개)은 수정**. 학과별 소개 *내용 풍부도*(AX는 course 위주)는 데이터/청킹(P4) 과제로 보류.

### D8. 버그 A 원인 추적: 단일의도붕괴 → content_type오염(틀림) → 키워드LIKE오염(맞음)
- 1차 의심: per-task 복사 시 content_type 오염 → 고쳤으나 여전히 person 0행.
- 추적: person 테이블이 질문에서 뽑은 단어를 name/email에 **LIKE 필터**로 거는데, 다중 의도 질문의 "담당/과목"이 person을 0행으로 만듦.
- → 다중 의도 질문에선 **키워드 LIKE 필터를 끄고 dept만** 사용(키워드를 어느 테이블에 귀속할지 모호하므로).

### D9. 근거검사: 주 의도만 vs 의도 집합 합집합 → **합집합(부분 답변 허용)**
다중 의도에서 한 의도(person)만 근거가 있어도 답을 막지 않고, 있는 것(이메일)은 주고 없는 것(과목)은 정직하게 "자료 없음"으로.

### D10. 브랜치: main 직접 vs 별도 → **`KSJ_4th`**
사용자 요청으로 main을 보호하기 위해 별도 브랜치에서 작업.

### D11. 마무리 범위: 검증된 버그만 vs 리팩터링 강행 → **검증된 버그까지(현재)**
P2 책임분리 리팩터링과 S5 점수제 정교화는 사용자 체감 없는 품질/조건부 작업이라, "측정이 요구할 때/여력 있을 때"로 보류. (아래 5절)

### D12. (재검토) 내 수정의 과잉트리거 발견 → 접속어 게이트로 보수화
"한 번 더 검토" 과정에서 **내가 추가한 다중의도 탐지가 너무 헐거웠음**을 발견. "입학 일정"처럼
'일정'(event 키워드)을 포함한 흔한 단일 질문이 admission+event로 잘못 잡혀 event SQL이 섞이고
키워드 필터까지 꺼졌다. → 보조 의도를 **명시적 접속어가 있을 때만** 채택하도록 보수화(2b70cad).

## 7. 작업 방식 교훈
1. 인수인계 진단은 **증상과 원인을 분리 검증** (P3는 "위치설" → 실제는 "룰 순서 + 단일 의도 한계").
2. **자명해 보이는 한 줄 수정이 no-op일 수 있다** (P1-b: 빈 매핑 fallback / 검색기 선필터).
3. **버그 판정은 중간 신호 말고 최종 답변으로** — 추론으로 "no-op/버그" 결론을 내렸다가 실측에서 뒤집힌 일이 반복됐다.
4. **권장 내기 전 데이터부터, 도구 바꾸기 전 라벨부터** (S3 등급제 시도가 데이터로 무산, 검증 하네스는 재설계 대신 라벨 큐레이션으로 충분).
