## 1. 인수인계 5개 의심 버그 — 검증 결과

| # | 주장 | 판정 | 근거 |
|---|---|---|---|
| P1-a | 질문이 2회 분석됨 | **무해(격하)** | `analyze()`는 순수함수라 2번 호출해도 결과 동일. 성능 비용도 무시 가능 |

| P2 | 분류기가 너무 많은 책임을 가짐 | 사실(리팩터링 가치 O) | 미착수(유지보수용, 선택) |

## 4. 의사결정 기록 (왜 이렇게 했는가)

> 이 작업은 갈림길이 많았다. "쉬운 길"이 아니라 "데이터/실측이 지지하는 길"을 고르려고 했고, 그 과정에서 몇 번 방향을 바꿨다. 팀이 나중에 "왜 이렇게 했지?"를 이해할 수 있도록 기록한다.

### D1. 작업 방식: 추측 수정 vs 검증 후 수정 → **검증 후**
인수인계 코멘트가 전부 "~인 것 같다" 톤이라, 각 주장을 코드/실측으로 참·거짓 판정한 뒤에만 손댔다. 그 결과 의심 버그 5개 중 3개가 무해로 판명되어 **불필요한(심지어 유해한) 수정을 피했다.**

### D2. P1-a(분석 2회): 긴급 버그 vs 무해 청소 → **무해로 격하**
파이프라인과 VectorRetriever가 `analyze()`를 각 1회씩 부르는 건 사실. 그러나 `analyze()`는 (질문, 직전학과)의 순수함수라 두 결과가 항상 동일 → 정합성 버그 아님. "성능 버그"라는 인수인계 프레이밍은 과장이었다. 우선순위 최하로 내림.

### D3. S3 근거게이트: 핵심 의사결정 (여러 번 방향 전환) → **결국 DROP**
- 처음엔 "일반/학과소개도 근거검사를 거치게 하자"는 인수인계 지적을 받아 **등급제 게이트**를 설계하려 했다.
- 사용자가 "쉬운 길 말고 올바른 길"을 요구 → 데이터를 직접 확인.
- **발견 1**: "학과 소개" 전용 content_type이 데이터에 없음(소개/overview 문자열 포함 문서 778개 중 2개) → 콘텐츠타입 기준(B안)은 **구현 불가**(과차단).
- **발견 2**: 검색기가 특정 학과 질문이면 이미 그 학과 문서만 통과시킴 → 학과-스코프 게이트(A안)는 **no-op**.
- **발견 3**: 실제로 학과 소개 답변을 생성해보니, content_type이 admission/course로 라벨링됐어도 내용은 **PDF 서술 기반으로 양호**했다. 즉 "소개가 조각으로 만들어져 나쁘다"는 전제 자체가 틀림.
- **결론**: 근거게이트 수정은 no-op이거나 거짓거부를 유발 → **S3와 그 대안(검색가점 S3′) 모두 폐기.** (P1-a처럼 "버그처럼 보였지만 무해")

### D4. S5 분류 엔진: 순수규칙 점수제 vs LLM vs 하이브리드 → **순수규칙(결정적), 격상은 측정 조건부**
- LLM 분류기는 의미 이해엔 강하나 비결정적이라 **오프라인 검증 하네스를 깨고** 매 질문 지연·비용·인젝션 표면을 늘림.
- 측정된 오분류는 전부 결정적 규칙으로 해결 가능했고, LLM이 *반드시* 필요한 의미 실패는 측정되지 않음.
- → 1차는 결정적 규칙. 임베딩/LLM 격상은 "골든이 잔여 의미오류를 숫자로 보여줄 때"만. (검증 안 된 단계에서 LLM 박는 것도 게으름의 반대 버전)

### D5. 다중 의도 범위: (a) 최소 2-의도 vs (b) N-일반 → **모델·배관은 N-일반 한 번에, 분석기 채우기는 보수적**
- (a)를 2-의도 특수 케이스로 하드코딩하면 나중에 (b)에서 통째로 들어내야 함(재작업 2번).
- 변경 비용의 대부분은 "스칼라→집합" 배관 공사인데 그건 2개든 N개든 한 번 들어감.
- → **데이터모델·배관은 N-일반으로 한 번에**, 분석기는 명확한 공동 의도만 보수적으로 채움. 나중에 일반화(b)는 **분석기 한 곳만** 손보면 됨(재작업 0).

### D6. 골든 큐레이션: 코드에 맞춰 통과시키기 vs 근거별 판정 → **근거별 판정**
"코드 출력대로 골든을 바꿔 통과시키기"(teaching to test)는 거부. 11개 실패를 (가)낡음 / (나)실버그 / (제품결정)으로 분류해, 낡은 건 근거 있게 정정하고, 진짜 결함은 그대로 두고, 입학 라우팅(sql-first vs vector)은 답변 품질 측정 전까지 **OPEN**으로 보류.

### D7. 버그 B 원인: 라우팅 vs 검색커버리지 vs 데이터 → **검색커버리지로 수정, 내용은 P4 보류**
- 처음엔 #5 라우팅(sql vs hybrid) 문제로 의심.
- 추적: SQL `department` 테이블이 비어 둘 다 vector로 답함 → 라우팅 무관.
- 진짜 원인: scope 질문의 vector 커버리지(전역 검색이 AX를 놓침).

### 8.0 발단: 노트의 "46명 표"가 거짓이었다
- 재베이스라인: `validate_rag_quality.py` **40/40**, `sql_multi_intent_smoke.py` **ALL PASS**, 버그A(`@` 포함)·버그B(4학과) 답변레벨 재확인 — 여기까진 노트와 일치.
- 그런데 **버그A 답변 전문을 처음으로 끝까지** 봤더니 교수 이메일 표가 **30행**뿐이었다. people.csv의 AI시스템학과 교수는 **46명**.
- 즉 2·3절의 "교수 이메일 46명 표"는 **raw SQL 행수(46)** 만 보고 쓴 것이고, **실제 렌더된 답변(30)** 은 확인된 적이 없었다. 우리 자신이 [[feedback_verify_at_output_level]]을 어긴 사례.

### 8.1 원인 추적 — 가설을 측정이 두 번 뒤집음
1. **단계별 행수 계측**: `people.csv 46` = `SQL raw 46` = **`built_context 30`** = `answer 30`. → 절단 지점은 LLM이 아니라 **context_builder**(LLM은 받은 30을 충실히 렌더, `answer ⊆ context` 성립).
2. **1차 가설 "행 캡(`max_sql_rows=30`)"** → 측정으로 반박: `max_total_context_chars=12000`인데 30행이 이미 **11,838자**. 행 캡과 문자 캡이 ≈같은 지점에서 만나, 행 캡만 올려도 문자 캡이 다시 30행으로 자른다(무효).
3. **진짜 원인 = 컬럼 부풀림**. person을 `SELECT *`로 18컬럼(`image_url`·`source_url`·`crawled_at`·`missing_fields`·`*_id`·코드중복 `dept` 등 메타 다수) 실어 행당 ~390자. 측정: 46행×18컬럼 = **21,819자**(→12000 초과로 30행 절단), 메타 제거 시 46행×12컬럼 = **5,293자**(여유). **근본 원인은 행이 아니라 컬럼이었다.**

### 8.2 수정 내역 (A~D, 파일 4개)
- **A. 컬럼 프루닝** (`context_builder.py`) — LLM에 무용한 메타/식별자 컬럼을 컨텍스트 표에서 제외. 모듈상수 `_SQL_CONTEXT_NOISE_COLUMNS`(`source_url`,`crawled_at`,`missing_fields`,`image_url`,`dept`,`raw_values`) + `record_id`/`*_id` 패턴(`_is_context_noise_column`). **범용 메타 블랙리스트**라 테이블별 분기(P2 악화) 없이 `course_name`/`title`/`content`/`email` 등 의미 컬럼은 전부 보존. raw rows/columns 원본은 불변(표시 단계에서만 프루닝) → 스모크/sources 영향 0. 재검토로 `_get_sql_columns` 외 사용처 없음 확인.
- **B. 행 캡 정합** (`context_builder.py`) — `max_sql_rows` 30→**100**(SQL단 `sql_tool.max_rows=100`과 정합). 행 캡은 상한일 뿐, 실질 절단은 C1의 문자 예산이 담당.
- **C1. 행 경계 절단 + 고지 보존** (`context_builder.py`) — 기존 `_limit_context`는 전체 문자열을 12000에서 **단순 슬라이스**해 마크다운 표를 행 중간에서 깨고 "총 N개 중 M개" 고지까지 날렸다. `_build_sql_context`가 **행을 추가하며 문자 예산을 행 단위로 적용**(최소 1행 보장, 고지용 여유 `_SQL_NOTICE_RESERVE_CHARS=90` 확보)하고, 잘리면 표 직후에 고지를 남기도록 변경. `build()`는 SQL을 vector보다 앞에 두고 표별로 남은 예산을 차감 → 최후 안전망 `_limit_context`가 잘라도 SQL 표·고지는 보존된다.
- **C2. 고지의 답변 반영** (`answer_generator.py`) — 시스템 프롬프트 **규칙 22** 추가: context에 "총 N개 중 M개만 표시" 절단 안내가 있으면 일부를 전부인 양 답하지 말고 절단 사실 + 좁혀 묻는 법을 답변에 명시.
- **D. 단일의도 person 0행 수정** (`sql_tool.py`) — "AX학과 교수님들 이메일 알려줘"류가 person **0행**(146명 중 vector로 5명만)이 되던 **선재 버그**. 원인: 호칭/목록 단어(`교수님들`,`명단`)가 `generic_words`에 없어 키워드로 살아남아 `name/email LIKE %교수님들%`→0행(MySQL 키워드절은 CSV `require_match=False`와 달리 빈결과 fallback이 없음). → person `generic_words`에 호칭(`교수님`,`교수님들`,`교원`), common에 목록어(`명단`,`명부`,`리스트`) 추가. **표적 수정**(전면 해제 아님) — 특정 이름(`김재경`)·연구분야(`로봇`) 키워드는 보존(D15).

### 8.4 (재검토) 내 수정의 결함 — **미해결 2건 + 한계 2건**
"멈추고 한 번 더" 재검토에서 발견([[feedback_operating_mode]]/D12 정신).
- **① 고지가 거짓 총계를 주장 (미해결, 심각)**: 고지 `총 N개 중 M개`의 `N = len(sql_result.rows)`인데 이건 **SQL `LIMIT 100`으로 이미 잘린 값**이다. AX 실제 146명인데 고지·답변은 "총 100명"이라 단정(C2가 증폭). baseline(고지 없음) 대비 후퇴는 아니나, **모르는 총계를 자신 있게 틀리게** 말하는 새 리스크. → 권고: `sql_tool`이 LIMIT 전 **진짜 COUNT**를 반환하게 하면 고지·답변이 정확해지고, 앞서 인정한 "SQL캡 침묵"(146→100)도 **동시에** 해결된다.
- **② 표별 예산 순서 의존·불공정 (미해결, 중간)**: `build()`가 첫 표에 예산을 몰빵하고 차감 → `_search_sql_all`이 먼저 주는 표가 독식. 실측: "AX학과 교수 이메일이랑 담당 과목"에서 **course(78 전부)가 person(45/100)을 굶긴다**(사용자가 "이메일"을 먼저 말해도). 정직하나(고지 있음) 배분이 자의적·순서의존. → 권고: 의도 우선순위 정렬 또는 예산 분할.
- **③ C1 O(n²) (한계, 경미)**: 행마다 `len("\n".join(lines))` 재계산. 기능은 정상, 누적 카운터로 정리 가능.
- **④ LLM 출력 변동 (한계, 코드 불가)**: 컨텍스트는 항상 46/46인데 LLM이 렌더 시 임의로 줄인다(동일 질문 4회: 46·46·46·**7**). 완전성 수정의 *실효*를 약화하나 생성 레이어라 코드로 보장 불가(프롬프트로 완화 시도 가능).
