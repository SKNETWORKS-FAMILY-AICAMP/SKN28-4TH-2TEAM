## 1. 프로젝트 개요

### 1.1 배경

KAIST AI 관련 학과(AI컴퓨팅학과, AI시스템학과, AX학과, AI미래학과)의 입학·교과목·교수진·사무실 등 정보는 여러 공식 페이지와 PDF 문서에 흩어져 있어, 지원자가 필요한 정보를 한 번에 찾기 어렵다. 본 프로젝트는 흩어진 내·외부 문서를 수집·전처리하고 LLM 기반 RAG(Retrieval-Augmented Generation) 파이프라인과 연동하여, 사용자의 자연어 질문에 정확한 근거와 함께 답변하는 웹 애플리케이션을 개발한다. 단순 질의응답을 넘어 사용자별 대화 기록, 답변 피드백, 운영 통계, 챗봇 보완 Q&A 게시판 등 서비스 운영 관점의 기능을 함께 제공한다.

### 1.2 목적

- 사용자가 KAIST AI 학과 정보를 대화형으로 질의하고, 출처가 포함된 답변을 얻을 수 있도록 한다.
- HTML/CSS/JavaScript 프론트엔드와 Django 백엔드로 구성된 웹 애플리케이션을 구축한다.
- 기존(3차)에 구현한 RAG 엔진을 웹 서비스 형태로 통합하고, 사용자·대화·피드백을 관리하는 운영 계층을 구현한다.
- 클라우드 환경에 인프라를 구성하고 애플리케이션을 배포한다.

### 1.3 범위

**포함 범위**

- 수집된 4개 KAIST AI 학과 데이터 기반 질의응답
- SQL(정형) + Vector(비정형) 하이브리드 검색 및 LLM 답변 생성
- 회원가입/로그인, 사용자별 대화 세션·기록 관리
- 답변 출처 기록·표시, 답변 피드백, 실패 질문 로그, 운영 통계
- 챗봇 보완용 Q&A 게시판
- 웹 채팅 UI, Django REST API
- 클라우드 인프라 구성 및 배포(최소 1회 시연 수준)

**제외 범위**

- 크롤링 문서 CRUD·업로드·권한·재색인 관리자 기능(데이터 파이프라인이 이미 처리)
- 수집하지 않은 KAIST 학과 및 KAIST 외 일반 질문에 대한 상세 답변
- 개인별 합격 여부·합격 가능성·선발 확률 등 예측성 답변(안전 정책상 차단)
- 결제 등 본 과제 범위를 벗어난 기능

> ※ 크롤링 데이터 관리는 MySQL·전처리·벡터화 파이프라인이 담당하고, Django는 사용자 서비스 운영 계층으로 사용한다. 클라우드 배포 수준(상시 운영 vs 시연용 1회 배포)은 강사 가이드 확인 후 팀에서 확정하며, 본 문서는 '시연용 배포'를 기준으로 한다.

## 2. 시스템 구성 개요

시스템은 프론트엔드, 백엔드(Django), RAG 엔진, 데이터 저장소(정형 DB·Vector DB), 외부 LLM API의 영역으로 구성된다.

| 구성 요소 | 역할 |
|-----------|------|
| 프론트엔드 | HTML/CSS/JavaScript 기반 채팅·게시판 화면. 질문 입력, 답변·출처 표시, 응답 스트리밍 렌더링 |
| 백엔드(Django) | REST API 제공, 인증/세션, RAG 파이프라인 호출, 대화·피드백·게시판·통계 관리 |
| RAG 엔진 | 질문 분석 → 검색(SQL/Vector) → context 구성 → LLM 답변 생성 (기존 `src/rag` 재사용) |
| 정형 DB (MySQL) | 입학·교과목·교수진·사무실 등 정형 데이터 저장 및 SQL 조회 |
| Vector DB (Chroma) | 문서 임베딩 저장 및 의미 기반 검색 |
| 외부 LLM API | OpenAI ChatOpenAI 기반 답변 생성, 임베딩 생성 |
| 클라우드 인프라 | 애플리케이션 서버 및 DB 호스팅, 배포 환경 |

> ※ 상세 시스템 구성도(다이어그램)는 별도 산출물 '시스템 구성도'에서 작성한다.

## 7. 제약사항 및 가정

- 프론트엔드는 HTML/CSS/JavaScript, 백엔드는 Django로 구현한다(가이드라인 명시).
- 회원가입/로그인은 교과과정 필수 항목으로 반드시 포함한다. Django 기본 인증을 사용하며 이메일 인증·소셜 로그인은 범위에서 제외한다.
- OpenAI API Key 및 사용 가능한 Chroma DB·MySQL이 준비되어 있어야 정상 동작한다.
- 답변은 수집된 4개 학과 데이터 범위로 한정되며, 범위 밖은 안내·거절한다.
- 클라우드 배포는 시연 가능한 최소 구성을 기준으로 한다(상시 운영 비용·범위는 별도 협의).
- LLM·임베딩은 외부 API에 의존하므로 네트워크·요금·응답시간의 영향을 받는다.
