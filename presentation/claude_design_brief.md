# [요청] 발표 슬라이드덱 + 스피커 노트 제작

아래 내용으로 **발표용 슬라이드덱(17장)** 을 만들어줘. 그리고 **각 슬라이드마다 발표자가 실제로 말할 스피커 노트(대사)** 도 함께 작성해줘.

## 제작 조건
- **언어/포맷**: 한국어, 16:9, 17장
- **발표 시간**: 20분 — 스피커 노트는 전체 합쳐 20분 말 분량(자연스러운 구어체), 슬라이드 간 전환 문장 포함
- **핵심 메시지**: 단순 모델 개선이 아니라 **데이터 → DB → 백엔드 → 프론트 → 배포 전 구간을 직접 연결한 "브릿지 프로젝트"**. 각 슬라이드 모서리에 다루는 **계층 태그**(데이터/DB/백엔드/프론트/배포)를 작게 표시.
- **디자인**: 마스코트 '넙죽이' + KAIST 블루 포인트, 깔끔하고 시각적. 한 슬라이드 한 메시지, 큰 헤드라인 + 여백, 다이어그램·수치 카드·스크린샷 강조.
- **청중**: 데모데이 평가자. 평가 산출물 5종(요구사항 정의서·화면설계서·LLM 연동 웹앱·시스템 구성도·테스트 계획·결과)이 자연스럽게 드러나게.
- **이미지**: 첨부한 스크린샷을 화면 관련 슬라이드(11·12번)에 배치. 나머지 다이어그램/구성도는 적절히 그려주거나 placeholder.
- **[채우기]** 표시된 두 곳(배포 환경·회고)은 **빈칸으로 남겨줘.** 나중에 직접 채울 예정.

## 슬라이드 내용

**1. 표지**
- 넙죽이 — KAIST AI·자연과학·생명과학기술대학 안내 RAG 챗봇
- 부제: 데이터부터 배포까지, 서비스 구조를 연결한 브릿지 프로젝트
- SKN28 4기 2팀 · 2026-06-26

**2. 프로젝트 정의**
- 주제: LLM을 연동한 내·외부 문서 기반 질의응답 웹 애플리케이션
- 한 줄: 흩어진 KAIST 학과 정보를 출처와 함께 대화로 답한다
- 이번 4차의 의미: 모델 한 조각이 아니라 전체 서비스 구조를 연결

**3. 팀 & 역할 (R&R)**
- 김성재 (발표): RAG · DB
- 신혜지: 백엔드 · 데이터
- 심기성: 프론트 · 배포
- 손지은: 프론트 · 데이터

**4. 배경 / 문제 정의**
- KAIST 3개 단과대 정보가 여러 공식 페이지·PDF에 흩어짐 → 한 번에 찾기 어려움
- 목표: 자연어로 묻고 근거(출처) 있는 답을 받는 서비스

**5. 요구사항 정의서**  `[평가: 요구사항]`
- 프로젝트 목표 + 사용자 요구사항 명확화
- 핵심 기능·서비스 흐름: 로그인 → 질문 → 출처 답변 → 게시판/문의 → 관리자
- 기능 / 비기능 요구사항 구분 (인증·채팅·검색·커뮤니티 / 보안·안정성·배포)
- 구현 범위·제약·우선순위를 일정에 맞춰 정리

**6. 시스템 구성도 (핵심)**  `[평가: 시스템 구성도]`
- 흐름: 크롤링·전처리 → MySQL + Chroma(벡터) → RAG 파이프라인 → Django API → 프론트 → 배포
- 사용자–프론트–백엔드–LLM–DB 관계와 질문→답변 데이터 흐름을 한 장에
- 사용 기술·각 구성요소 역할 + 로컬/배포 환경 포함
- (아키텍처 다이어그램 — 5개 계층 가로 연결)

**7. [데이터] 수집·전처리**
- 3개 단과대학 크롤링 → 정형(CSV)·비정형(문서) 분리, 품질 점검·중복 제거

**9. [백엔드/LLM] 하이브리드 RAG**  `[평가: LLM 연동 웹앱]`

- (질문→답변 시퀀스 다이어그램)
