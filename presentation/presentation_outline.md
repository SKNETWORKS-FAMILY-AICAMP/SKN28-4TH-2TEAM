# 넙죽이 발표 슬라이드덱 초안 (20분 발표)

> 클로드 디자인에 **PART A(프롬프트)** → **PART B(슬라이드 내용)** 순서로 붙여넣으면 됩니다.
> 서사: *"RAG 성능 고도화"가 아니라, 데이터–DB–백엔드–프론트–배포까지 **서비스 구조를 직접 연결한 브릿지 프로젝트"*

---

## PART A — 클로드 디자인에 줄 프롬프트

> KAIST 학과 안내 RAG 챗봇 **"넙죽이"** 데모데이 발표 슬라이드덱을 만들어줘.
> - **포맷**: 한국어, 16:9, **20분 발표용 약 17장**
> - **핵심 메시지**: 단순 모델 개선이 아니라 **데이터→DB→백엔드→프론트→배포 전 구간을 직접 연결한 "브릿지 프로젝트"**. 각 슬라이드 모서리에 지금 다루는 **계층 태그**(데이터/DB/백엔드/프론트/배포)를 작게 표시.
> - **디자인**: 넙죽이 마스코트 + KAIST 블루, 깔끔·시각적. 아키텍처 다이어그램·수치 카드·스크린샷 강조.
> - **청중**: 데모데이 평가자. **평가 산출물 5종**(요구사항 정의서·화면설계서·LLM 연동 웹앱·시스템 구성도·테스트 계획·결과)이 명확히 드러나게.
> - **스피커 노트**: 각 슬라이드마다 발표자가 실제로 말할 대사를 1~3문장으로 함께 작성해줘. 전체 합쳐 **20분 발표 분량**(말 속도 기준)으로, 담백하고 자연스러운 구어체. 슬라이드 간 전환 문장도 포함.
> - 아래 슬라이드 내용을 기반으로 디자인. 이미지(스크린샷·구성도)는 placeholder로 자리만.

### 5. 요구사항 정의서 · (1:30) `[평가①]`
- **프로젝트 목표 + 사용자 요구사항** 명확화
- **핵심 기능·서비스 흐름**: 로그인 → 질문 → 출처 답변 → 게시판/문의 → 관리자
- **기능 / 비기능 요구사항** 구분 (인증·채팅·검색·커뮤니티 / 보안·안정성·배포)
- **구현 범위·제약·우선순위**를 일정에 맞춰 정리
- *(요구사항 표 일부 캡처 — `docs/요구사항정의서.md`)*

### 6. ⭐ 시스템 구성도 — 전체 브릿지 · (2:00) `[평가④][브릿지]`
- `크롤링·전처리 → MySQL + Chroma(벡터) → RAG 파이프라인 → Django API → 프론트 → 배포`
- **사용자–프론트–백엔드–LLM–DB 관계**와 **질문→답변 데이터 흐름**을 한 장에
- 사용 기술·각 구성요소 역할 + 로컬/배포 환경 포함
- *(아키텍처 다이어그램 — 5개 계층 가로 연결)*

### 7. [데이터 계층] 수집·전처리 · (1:00) `[데이터]`
- 3개 단과대학 크롤링 → 정형(CSV)·비정형(문서) 분리, 품질 점검·중복 제거
- RAG 적재 포맷(jsonl)으로 변환

### 8. [DB 계층] 정형 + 벡터 이원화 · (1:00) `[DB]`
- MySQL `kaist_ai`(학과·교수·교과목) + Chroma 벡터스토어
- **7,759 chunks · 13개 학과 · 3개 단과대학** 커버

### 9. [백엔드/LLM 계층] 하이브리드 RAG · (1:30) `[백엔드][평가③]`
- **SQL(정형) + Vector(의미) 하이브리드** 검색 → 출처 카드로 근거 제시
- 질의 분석(학과·의도) → 검색 → LLM 답변 → 출처/피드백
- *(질문→답변 시퀀스 다이어그램)*

### 10. [백엔드] 측정 기반 개선 사례 · (1:30) `[백엔드]`
- **평가 하니스**(39문항)로 측정 → 병목은 검색이 아니라 **라우팅**임을 발견
- 답변율 **66.7% → 94.9%** / 출처 정확도 59% → 87.2% / 검색 hit@5 97.4% 유지
- *(전후 수치 카드)* — "추측이 아니라 수치로 개선"
